# Quickstart: Content Authoring Guide

**Feature**: 001-curriculum-restructure
**Date**: 2025-12-04
**Purpose**: Step-by-step guide for creating curriculum content following the 4-module, 13-week structure

## Overview

This quickstart provides templates, file naming conventions, and workflow steps for authoring Physical AI book content. Follow these guidelines to ensure consistency with Docusaurus autogenerated sidebars and constitutional requirements.

---

## Prerequisites

- Familiarity with Markdown/MDX syntax
- Understanding of YAML frontmatter
- Access to `docs/` directory in repository
- Read: `data-model.md` for entity structure
- Read: `research.md` for pedagogical best practices

---

## File Structure & Naming Conventions

### Module Files

**Location**: `docs/module-XX-<name>/index.md`

**Naming Pattern**:
- `module-01-ros2/index.md`
- `module-02-gazebo-unity/index.md`
- `module-03-nvidia-isaac/index.md`
- `module-04-vla-conversational/index.md`

**Purpose**: Module overview page with learning objectives, prerequisites, and week links

---

### Week Files

**Location**: `docs/module-XX-<name>/week-YY-<topic>.md`

**Naming Pattern**:
- `week-01-intro-physical-ai.md` (in module-01-ros2/)
- `week-06-gazebo-fundamentals.md` (in module-02-gazebo-unity/)
- `week-13-conversational-robotics.md` (in module-04-vla-conversational/)

**Rules**:
- Use lowercase with hyphens (kebab-case)
- Start with `week-YY-` where YY is 01-13
- Follow with descriptive topic name

## Content Authoring Workflow

### Step 1: Create Module Folder & Index

```bash
# From repository root
mkdir -p docs/module-01-ros2
touch docs/module-01-ros2/index.md
```

Fill `index.md` with Template 1 (Module Index)

### Step 2: Create Week Files

```bash
# Create all weeks for the module
touch docs/module-01-ros2/week-01-intro-physical-ai.md
touch docs/module-01-ros2/week-02-embodied-intelligence.md
touch docs/module-01-ros2/week-03-ros2-architecture.md
# ... etc.
```

Fill each week file with Template 2 (Week Content)

### Step 3: Verify Frontmatter

Check that:
- [ ] `sidebar_position` values are sequential and unique within module
- [ ] `week_number` matches file name (week-03 → week_number: 3)
- [ ] `module` field references correct module_id
- [ ] `hands_on_exercises` has at least 1 item (FR-015)

### Step 4: Add Code Snippets

Follow hybrid approach (from research.md):
- **Inline**: Core concepts (15-30 lines)
- **External**: Complete projects (link to GitHub)

Example:
```markdown
\```python
# Inline snippet: minimal ROS 2 publisher
import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class MinimalPublisher(Node):
    def __init__(self):
        super().__init__('minimal_publisher')
        self.publisher_ = self.create_publisher(String, 'topic', 10)
\```

**Complete working example**: [GitHub: ros2-minimal-publisher](https://github.com/example/ros2-examples)
```

### Step 5: Cross-Reference Prerequisites

Link to earlier weeks/modules explicitly:

```markdown
## Prerequisites

This week builds on concepts from:
- [Week 2: Embodied Intelligence](./week-02-embodied-intelligence.md)
- [Module 1 Overview](./index.md)
```

### Step 6: Test Autogenerated Sidebar

```bash
# Build Docusaurus to verify sidebar
npm run build

# Or serve locally
npm run start
```

Navigate to new module/week in browser to confirm:
- Sidebar displays module folders as categories
- Week files appear in correct order (by sidebar_position)
- Navigation works (Previous/Next buttons)

### Step 7: Validate Links

Docusaurus will throw errors on broken links (config: `onBrokenLinks: 'throw'`)

```bash
# Build to catch broken links
npm run build
```

Fix any errors before committing.

## Writing Guidelines

### Learning Objectives

Use action verbs (Bloom's Taxonomy):
- **Remember**: Identify, define, list, name
- **Understand**: Explain, describe, summarize, distinguish
- **Apply**: Implement, use, execute, demonstrate
- **Analyze**: Compare, contrast, examine, differentiate
- **Evaluate**: Assess, justify, critique, prioritize
- **Create**: Design, construct, develop, integrate

**Good**:
- "Implement a ROS 2 publisher node using rclpy"
- "Compare Gazebo and Unity simulation capabilities"

**Bad**:
- "Learn about ROS 2" (too vague)
- "Understand nodes" (not measurable)

---

### Code Snippet Best Practices

1. **Keep it concise**: 15-30 lines for inline snippets
2. **Add comments**: Explain non-obvious logic
3. **Use syntax highlighting**: Specify language (```python, ```bash)
4. **Link to full projects**: GitHub repos for 50+ line projects
5. **Test all code**: Ensure snippets actually run

---

### Hands-On Exercises

Each week MUST have ≥1 exercise (FR-015):

**Structure**:
- **Title**: Clear, action-oriented
- **Description**: What learners will accomplish
- **Tools Required**: Specific versions (e.g., ROS 2 Humble, Python 3.10+)
- **Estimated Time**: Realistic (15 min - 2 hours)
- **Steps**: Numbered, detailed instructions
- **Expected Outcome**: Success criteria

**Example**:
```markdown
### Exercise: Create a ROS 2 Talker Node

**Objective**: Build a Python node that publishes String messages to a topic.

**Tools Required**:
- ROS 2 Humble
- Python 3.10+
- Ubuntu 22.04 or Docker

**Estimated Time**: 45 minutes

**Steps**:
1. Create a new ROS 2 workspace: `mkdir -p ~/ros2_ws/src`
2. Generate a Python package: `ros2 pkg create --build-type ament_python talker_pkg`
3. Implement the publisher node in `talker_pkg/talker_node.py`
4. Build and source: `colcon build && source install/setup.bash`
5. Run the node: `ros2 run talker_pkg talker_node`

**Expected Outcome**: Node publishes "Hello, ROS 2!" messages at 1 Hz.

**Complete Code**: [GitHub: ros2-talker-example](https://github.com/example/ros2-talker)
```

## FAQ

**Q: Should I manually update sidebars.ts?**
A: No. Sidebars are autogenerated (type: 'autogenerated'). Just follow the folder structure and sidebar_position values.

**Q: How do I handle images/diagrams?**
A: Place images in `static/img/` and reference with `![Alt text](/img/module-01/diagram.png)`

**Q: Can I use JSX/React components in content?**
A: Yes, Docusaurus supports MDX. But keep it simple—most content should be plain Markdown.

**Q: What if a week covers multiple modules' concepts (e.g., Week 1-2 intro before ROS 2)?**
A: Place foundational weeks (1-2) in Module 1's folder. They're conceptually part of the progressive learning path starting with ROS 2.

**Q: How do I test the curriculum locally before committing?**
A: Run `npm run start` to serve locally, navigate to new modules/weeks, verify navigation, links, and rendering.
